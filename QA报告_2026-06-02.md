1. **`mutual_aid_subscriptions` 缺 `radius_km` 列**
   - 现象：sos-service 通知附近救援者时 `.select('user_id, radius_km')` 命中不存在的列 → PostgREST 报错 → 查询返回空 → **触发 SOS 后"附近互助者"收不到任何通知/推送**。
   - 位置：`functions/sos-service/index.ts`（notifyRescuersInternal）、`functions/mutual-aid/index.ts`（subscribe/getNearbySOS）。
   - 修复：迁移第 1 段 `ADD COLUMN radius_km INTEGER DEFAULT 5`。

2. **`subscriptions.status` CHECK 不含苹果通知会写的状态**
   - 现状 CHECK：`active/expiring/inactive/cancelled/refunded`（migrations/20260425）。
   - apple-iap 写：`expired`（EXPIRED）、`billing_issue`（续费失败）、`revoked`（REVOKE）→ **这三类通知的状态更新被 CHECK 拒绝**，订阅生命周期状态不准。
   - 修复：迁移第 2 段放宽 CHECK 纳入这三个值。

> 做法：Supabase → SQL Editor → 粘贴 `20260602_qa_fixes.sql` 全文 → Run。零风险（加列 IF NOT EXISTS + 重建 CHECK）。

4. **同城互助：前端直连表、根本没调 mutual-aid 边缘函数**
   - 证据：`src/pages/MutualAid.tsx` 用 `supabase.from('mutual_aid_subscriptions').insert/update`（:193 等）和 `supabase.from('mutual_aid_responses').insert`（:211）直接读写；全仓库无 `invoke('mutual-aid')`。
   - 后果：
     - ① **互助积分（total_rewards / 排行榜）其实从没被发放**——发分逻辑（REWARD_POINTS / addRewardPoints）只在边缘函数里，而前端不调它。
     - ② 我加的**防刷状态机 + GPS 到场核实 + 鉴权**对前端这条真实路径**不生效**（边缘函数没人调）。
     - ③ 响应失败无任何提示（`MutualAid.tsx:211` 裸 insert，无 try/catch、无结果检查）。
   - 建议（二选一，需你定）：
