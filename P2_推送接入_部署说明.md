## 二、你需要准备的凭据

### A. iOS（APNs）— 你已有开发者账号 ✅
1. Apple Developer → Certificates,IDs&Profiles → **Keys** → 新建 Key，勾选 **Apple Push Notifications service (APNs)** → 下载 `.p8`（**只能下一次**）。记下 **Key ID**；Team ID 在账号右上角。
2. Xcode 打开 `ios/App` → Target → Signing&Capabilities → **+Capability → Push Notifications**；确认 Background Modes 勾了 Remote notifications（Info.plist 已配）。
3. **⚠️ 核对 Bundle ID**：`capacitor.config.json` 里现在是 `com.warrescue.appname`，**像占位符**。必须改成你在 App Store Connect 注册的真实 App ID，并保证三处一致：`capacitor.config.json` 的 appId、Xcode 的 Bundle Identifier、下面的 `APNS_BUNDLE_ID`。

Secrets：
```bash
supabase secrets set APNS_KEY_ID=XXXXXXXXXX
supabase secrets set APNS_TEAM_ID=YYYYYYYYYY
supabase secrets set APNS_BUNDLE_ID=com.your.realbundleid
supabase secrets set APNS_AUTH_KEY="$(cat AuthKey_XXXXXXXXXX.p8)"
# 走 TestFlight/真机调试用沙盒：临时加 APNS_USE_SANDBOX=true；上架正式包请去掉
```

### B. Android（FCM）— 需要一个 Firebase 项目
1. [Firebase 控制台](https://console.firebase.google.com) 建项目 → 添加 Android 应用（包名与 `capacitor.config.json` appId 一致）→ 下载 **`google-services.json`** 放到 `android/app/`（构建时自动启用推送；缺它则安卓推送不工作）。
2. 项目设置 → **服务账号** → 生成新的私钥 → 下载服务账号 JSON。

Secrets：
```bash
supabase secrets set FCM_SERVICE_ACCOUNT="$(cat service-account.json)"
# FCM_PROJECT_ID 一般能从 JSON 的 project_id 自动读出，无需单独设
```

> 没有 Firebase 也能先只上 iOS：FCM 密钥不配时，安卓推送会自动跳过，不影响 iOS。

## 三、部署
```bash
# 1) 迁移：建 device_tokens 表
supabase db push

# 2) 设好上面的 secrets

# 3) 部署函数（ai-alert 引用了 _shared/push.ts，需一并重部署）
supabase functions deploy push-dispatch
supabase functions deploy ai-alert

# 4) 同步原生工程并重新出包
npx cap sync
# iOS 用 Xcode Archive；Android 用 Android Studio 出 AAB
```

## 四、验证（真机）
1. 真机装上新包 → 登录 → 同意通知权限 → 查 `device_tokens` 表是否多出本机 token（platform 正确）。
2. 手动发一条测试推送：
```bash
curl -X POST "https://<PROJECT_REF>.supabase.co/functions/v1/push-dispatch" \
  -H "Authorization: Bearer <SERVICE_ROLE_KEY>" -H "Content-Type: application/json" \
  -d '{"userIds":["<你的user_id>"],"title":"测试预警","body":"这是一条推送测试","severity":"red"}'
# 期望返回 { success:true, sent:1, failed:0, ... }
```
3. **把 App 完全退到后台或锁屏**，再发一次 → 应能在锁屏收到横幅+声音（这就是 P2 的核心目标）。
4. 触发一次真实采集（P1 的 collect）→ 确认有新预警时自动推送到真机。

## 五、Critical Alerts（穿透静音/勿扰）—— 需 Apple 审批后再开
红色空袭预警最好能无视静音/勿扰。这需要：
1. 向 Apple 申请 `com.apple.developer.usernotifications.critical-alerts` 专项授权（[申请表](https://developer.apple.com/contact/request/notifications-critical-alerts-entitlement/)，公共安全类 App 有合理理由）。
2. 获批后：取消 `App.entitlements` 里那两行注释；告诉我，我再把权限请求里的 `criticalAlert` 选项与红色预警的 `critical` 声音补上（约 10 行改动）。
> 在获批前**不要**加该 entitlement，否则会导致代码签名失败。

## 六、已知限制 / 后续
- 目前预警会推给**所有**开启推送的用户，**尚未按地理位置过滤**（应按用户 `monitor_radius_km` + 坐标筛选）——建议作为紧接着的优化项。
- **SOS 的短信/电话/推送触达在 P3**（会复用本模块的 `sendPushToUsers`，并接入 Twilio）。
- 推送声音目前用系统默认；如需专用空袭警报声，需把音频文件加入 iOS bundle 与 Android `res/raw` 并在渠道/payload 指定。
