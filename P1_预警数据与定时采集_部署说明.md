# P1：真实预警数据接入 + 定时采集（部署与接入说明）

> 目标：让"预警"从随机假数据变成真实、可定时采集的数据，并修复历史上让插入直接失败的表结构错配。这是上架的第一块基石。

---

## 一、本次改了什么（代码侧，已完成）

1. **`functions/ai-alert/index.ts`** —— 设为唯一真实采集器
   - 默认 `collect` 不再生成假数据，而是从真实上游源**增量采集**（ReliefWeb / GDACS / ACLED / 以色列 Oref / Ukraine LiveUAMap）。
   - 假数据被锁进 `ALERT_DEMO_MODE=true` 开关，仅供本地测试 / 应用审核演示。
   - 列名对齐真实表：`alert_type`（不再写不存在的 `type`），去掉不存在的 `is_active` / `expires_at`。
   - **活跃判定统一为 `end_time IS NULL`**（与前端 Dashboard 查询一致）；过期由 `action=process` 写 `end_time` 实现。
   - 新增去重插入助手（按 `source_id` → `source_url` 去重）。

2. **`functions/monitor/index.ts`** —— 旧的重复采集器，降级为"仅 DEMO"
   - 生产模式下直接返回空并提示改用 ai-alert，**不再写入假/坏数据**；同样修正了列名与 `end_time` 判定。

3. **`migrations/20260529_alert_pipeline_realdata.sql`**（新）
   - 幂等补齐 alerts 缺失列：`confidence` / `verification_confidence` / `verification_notes`，并兜底补 `source_id/source_url/source_published_at/detected_at/detection_delay_seconds`。
   - 索引：活跃预警索引、`source_id` 唯一索引（去重）、`created_at` 索引。
   - 用 **pg_cron + pg_net** 启用定时任务（原迁移里这段是注释掉的）：实时采集每 2 分钟、过期处理每 5 分钟、清理每天一次。

4. **后台调用修正**：`admin/.../SystemMgmt.tsx`、`src/pages/AdminDashboard.tsx` 里指向旧 `monitor` 的"立即采集"按钮，全部重定向到 `ai-alert`；无效的 `action=check` 改为 `stats`；活跃数统计与"启用/停用预警"改为基于 `end_time`。

> 前端 `npm run typecheck` 已通过。

## 二、你需要提供/设置的（按必需程度）

### A. Supabase 函数环境变量（Edge Function Secrets）
| 变量 | 必需性 | 说明 |
|---|---|---|
| `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` | 必需 | Supabase 自动注入，一般无需手动设 |
| `MEOO_PROJECT_API_KEY` | 可选 | AI 分析器密钥；不填则跳过 AI 源，不影响真实源采集 |
| `ALERT_DEMO_MODE` | 可选 | **生产环境务必不要设为 true**；仅审核演示时临时打开 |
| `CRON_SECRET` | 建议预设 | cron 调用携带的密钥；强制校验随 P4 启用，先设好 |

设置示例：
```bash
supabase secrets set MEOO_PROJECT_API_KEY=xxxx
supabase secrets set CRON_SECRET=$(openssl rand -hex 24)
```

### B. 真实数据源（决定"预警有多真、多快"）
| 数据源 | 是否要 key | 现状 |
|---|---|---|
| **ReliefWeb** | 免费、无需 key | ✅ 开箱可用（联合国报告级，偏慢，覆盖广） |
| **GDACS** | 免费、无需 key | ✅ 开箱可用（灾害事件，非空袭专用） |
| **ACLED** | **需注册拿 key + email** | ⚠️ 没 key 该源会跳过/报错（不影响其它源） |
| 以色列 Oref | 公开端点 | ⚠️ 可能需特定请求头/可能被地理封锁，需实测 |
| Ukraine LiveUAMap | 非官方端点 | ⚠️ 可能限流/结构变动，不稳定 |

**强烈建议**（这才是"防空预警"最核心、最及时的源）：
- **乌克兰**：注册 [ukrainealarm.com 官方 API](https://api.ukrainealarm.com)（免费 token），秒级空袭警报。
- **以色列**：Pikud HaOref（Home Front Command）官方实时端点。

把这两个官方源接上，App 才真正具备"秒级空袭预警"能力。告诉我你拿到哪些 token，我来把对应源接入并按真实响应调好字段解析。

## 三、启用定时采集（二选一）

### 方式 1：pg_cron（推荐，已写进迁移）
在 Supabase **SQL Editor** 执行一次（把 `<...>` 换成真实值，**不要提交进仓库**）：
```sql
alter database postgres set app.settings.functions_url = 'https://<PROJECT_REF>.supabase.co/functions/v1';
alter database postgres set app.settings.cron_secret   = '<与 CRON_SECRET 相同的随机串>';
```
然后应用迁移 `20260529_alert_pipeline_realdata.sql`（见下）。

### 方式 2：Supabase Dashboard → Database → Cron / Scheduled Functions
图形界面新建任务，每 2 分钟 POST `…/functions/v1/ai-alert?action=realtime_monitor`。

## 五、验证（部署后请按此自测）
```bash
# 手动触发一次真实采集（不开 DEMO）
curl -X POST "https://<PROJECT_REF>.supabase.co/functions/v1/ai-alert?action=collect" \
  -H "Authorization: Bearer <ANON_OR_SERVICE_KEY>"
# 期望返回 { success:true, demoMode:false, collected:N, new:M, sources:[...] }
```
然后：
1. 查 `alerts` 表是否有新行、`alert_type/severity/source/source_id/end_time` 是否正常。
2. 查 `alert_crawl_state` 表，看各源 `items_found` 与 `error`。
3. 打开 App Dashboard，活跃预警应能展示（`end_time IS NULL` 的记录）。
4. 等几分钟确认 cron 自动在跑（`alert_crawl_state.last_crawl_at` 持续更新）。
